# 커밋 메세지 작성 방법  

  
# 1. 사용하는 이유 
팀원간 소통을 위해서    
  
내 수정 기록을 기억하기 위해서  
# 2. 커밋 메세지 구조 
```
Type(scope): Subject - Header  
Body 
Footer
```
## 2.1 Type 
Type에는 어떤 작업을 했는지 작성해준다.  
Type의 종류 
* feat: 새로운 기능 추가
* fix: 버그 수정
* docs: 문서 수정 (README.md 등)
* style: 코드 포맷팅, 세미콜론 누락 등 (코드 로직 변경 없음)
* refactor: 코드 리팩토링(품질 개선)
* test: 테스트 코드 추가 및 수정
* chore: 빌드 업무 수정, 패키지 매니저 설정 등 (기타 작업)
  
보통 feat이나 fix를 가장 많이 사용한다.

## 2.2 scope, Subject
### scope
어느 부분에 영향을 미쳤는지 Docs(README.md) 와 같이 작성  
### Subject
어떤 내용을 수정하였는지 작성  
50글자 내외로 작성, 마침표 없이 작성 Body(본문)과 구분 하기 위해서 한줄 띄어서 작성 
  
## 2.3 Body
필수로 작성하지는 않음  
무엇(What)을 왜(why) 수정 했는지 작성  
설명뿐만 아니라 커밋의 이유를 작성할 때도 작성
## 2.4 Footer
이슈 추적 ID 작성 할 때 사용 - 필수는 아님    
해결: #123   
관련: #124  
참고: #125

내가 1달 뒤 다시 보았을 때 이해할 수 있도록 Commit message를 작성해야 다른 사람이 보았을 때 이해할 수 있다.  

좋은 Commit message를 작성할 수 있도록 매 Commit message를 잘 작성해보자
